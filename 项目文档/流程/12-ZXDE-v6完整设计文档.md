## 一、项目目标

**输入**：主诉（症状）+ 脉象
**输出**：推荐方剂 + Top 5 候选 + 模块归属 + 命中症状

**核心价值**：用张锡纯本人的医案证据，量化方剂-症状-脉象关系，辅助临床决策。

## 四、v6 引擎设计

### 4.1 核心思路

**症状-方剂投票**：
- 对 535 方剂的 1102 案 C 字段聚合，建立"症状词 → 方剂名 → 频次"反向索引
- query 解析时：先做**同义词归并**（短气↔气短、胸闷↔闷），提取主症状词
- 对每个 query 症状词，查询其指向的所有方剂，按 `c / log(2 + n_cases)` 加权投票
- 对每个 query 脉象词，权重 0.3 做轻量投票

**模块先验**（30 个症状 → 模块归属）：
- 例：短气 → [气虚虚劳, 胸腹升降, 痰饮肺]
- 主模块 +1.0，次模块 +0.3

**症状硬规则**（7 条领域知识）：
- 脑充血中风：头晕+弦脉 → 镇肝熄风汤等 +5
- 少阴伤寒：肢厥+沉微脉 → 通脉四逆汤等 +4
- 女科崩漏：崩漏/出血 → 固冲汤/清带汤 +4
- 阳明热盛：发热+洪脉 → 白虎加人参汤等 +4
- 痰饮咳嗽：痰+咳嗽 → 理饮汤等 +3
- 脾胃泄泻：泄泻/腹胀 → 薯蓣粥等 +2.5
- 大气下陷：短气/胸闷 → 升陷汤等 +3

**log(2+n) 缩放**：避免"超级方"（n=20 案）压制小方剂（n=1 案），保护长尾方剂。

### 4.2 算法伪代码

```
for each 方剂 in 535方剂:
    score = 0
    for each query症状 in query.症状:
        for each (方剂, 频次) in 反向索引[query症状]:
            score += 频次 / log(2 + 方剂.案数)
        for each 同义词 in 同义词组[query症状]:
            score += 0.5 * 反向索引[同义词][方剂] / log(2 + 方剂.案数)
    for each query脉象 in query.脉象:
        score += 0.3 * 频次 / log(2 + 方剂.案数)
    for each 症状 in query.症状:
        if 方剂.模块 == MOD_PRIOR[症状][0]:
            score += 1.0
        elif 方剂.模块 in MOD_PRIOR[症状][1:]:
            score += 0.3
    for each 规则 in SYMPTOM_RULES:
        if 规则.match(query):
            for each boost方剂 in 规则.boost:
                if boost方剂 == 当前方剂:
                    score += 规则.score
sort(score) → Top 5
```

## 八、v6 引擎局限性

1. **症状词典规模**：仅 150 症状词，张锡纯原文中很多变体未收（如"胸中闷" vs "胸闷"）。可通过增加同义词组改善。
2. **方剂聚合**：仅用 C 字段症状计数，忽略症状的位置信息（出现在案首 vs 中段意义不同）。
3. **n=1 方剂**：单案方剂在投票中天然弱势（log 缩放后），但很多张氏代表方就是 1 案。
4. **未利用辨证**："气虚/阴虚/阳虚"等辨证词未进入投票。

## 九、后续优化方向

1. **症状词典扩展**：基于 994 案原文，自动挖掘新症状词。
2. **辨证字段加权**：P3 辨证整合加入投票。
3. **案首句/案尾句位置权重**：案首的症状比案中描述更确定。
4. **脉位 / 脉势 / 至数 全维度加入**：当前只用 N3c 脉形。
5. **多层投票**：症状 + 脉象 + 辨证 + 用药，四层加权。
6. **反馈学习**：用真实盲测 + 临床反馈更新权重。
